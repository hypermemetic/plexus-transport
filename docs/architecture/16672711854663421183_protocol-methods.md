## Overview

Plexus supports **two categories of methods**:

1. **User-defined methods** - Business logic defined with `#[hub_method]`
2. **Protocol-level methods** - Built-in discovery and introspection

Protocol methods provide universal capabilities across all activations:
- **Schema discovery** - Introspect available methods and their parameters
- **Cache validation** - Check if cached schemas are still valid
- **Backend discovery** - List available activations and server info
- **Generic invocation** - Single endpoint for calling any method

## Protocol Method Types

### 1. `schema` (Per-Activation)

**Purpose:** Schema introspection and discovery
**Scope:** Per-activation
**Available In:** All transports

```rust
// Usage
POST /rest/users/schema
{}
// Returns full PluginSchema

POST /rest/users/schema
{"method": "get_user"}
// Returns specific MethodSchema
```

**Unique Characteristics:**
- **Auto-generated** by macro as a regular method
- **Appears in `PluginSchema.methods[]`**
- Has its own `MethodSchema` with params, description, http_method
- Treated uniformly by transport bridges

**Implementation:**
```rust
// plexus-macros/src/codegen/method_enum.rs:301-307
let schema_method = MethodSchema::new(
    "schema".to_string(),
    "Get plugin or method schema. Pass {\"method\": \"name\"} for a specific method.",
    "auto_schema".to_string(),
);
methods.push(schema_method);  // ← Added to methods array
```

---

### 2. `{method}.schema` (Per-Method)

**Purpose:** Syntactic sugar for per-method schema lookup
**Scope:** Per-method
**Available In:** REST HTTP, WebSocket JSON-RPC

```rust
// Equivalent calls
POST /rest/users/get_user.schema
{}

POST /rest/users/schema
{"method": "get_user"}
```

**Implementation:**
```rust
// plexus-macros/src/codegen/activation.rs:264-275
_ if method.ends_with(".schema") => {
    let method_name = method.strip_suffix(".schema").unwrap();
    let plugin_schema = self.plugin_schema();
    if let Some(m) = plugin_schema.methods.iter().find(|m| m.name == method_name) {
        let result = SchemaResult::Method(m.clone());
        return Ok(wrap_stream(
            futures::stream::once(async move { result }),
            concat!(namespace, ".method_schema"),
            vec![namespace.into()]
        ));
    }
}
```

### 3. `{namespace}.call` (Generic Invocation)

**Purpose:** Generic method invocation subscription
**Scope:** Per-activation
**Available In:** WebSocket JSON-RPC only

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "users.call",
  "params": {
    "method": "get_user",
    "params": {"user_id": "123"}
  }
}
```

**Why WebSocket-only:**
- REST/MCP have direct method routing (`POST /rest/users/get_user`)
- WebSocket benefits from single subscription point
- Enables dynamic method discovery without reconnecting

**Implementation:**
```rust
// plexus-core/src/plexus/plexus.rs:762-778
module.register_subscription(
    "{namespace}.call",
    PLEXUS_NOTIF_METHOD,  // "result"
    "{namespace}.call_unsub",
    move |params, pending, _ctx, _ext| {
        let method: String = params.get("method")...;
        let method_params: Value = params.get("params")...;

        let stream = plexus.call(&method, method_params).await?;
        pipe_stream_to_subscription(pending, stream).await
    }
)
```

---

### 4. `{namespace}.hash` (Cache Validation)

**Purpose:** Get composite hash for cache invalidation
**Scope:** Per-activation
**Available In:** WebSocket JSON-RPC only

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "users.hash"
}

// Response
{
  "jsonrpc": "2.0",
  "method": "result",
  "params": {
    "content": {"hash": "a1b2c3d4e5f6..."}
  }
}
```

**Use Case:**
- Client caches `PluginSchema`
- Periodically checks `.hash` to detect changes
- Re-fetches schema only if hash changed

**Implementation:**
```rust
// plexus-core/src/plexus/plexus.rs:780-797
module.register_subscription(
    "{namespace}.hash",
    PLEXUS_NOTIF_METHOD,
    "{namespace}.hash_unsub",
    move |_params, pending, _ctx, _ext| {
        let schema = Activation::plugin_schema(&plexus);

        let hash_stream = futures::stream::once(async move {
            json!({ "hash": schema.hash })
        });

        pipe_stream_to_subscription(pending, wrap_stream(...)).await
    }
)
```

### 5. `_info` (Backend Discovery)

**Purpose:** Server and activation discovery
**Scope:** Global (not per-activation)
**Available In:** WebSocket JSON-RPC only

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "_info"
}

// Response
{
  "backend": "my-server",
  "activations": [
    {
      "namespace": "users",
      "version": "1.0.0",
      "description": "User management"
    },
    {
      "namespace": "posts",
      "version": "1.2.0",
      "description": "Post management"
    }
  ]
}
```

**Use Case:**
- Client connects to server
- Calls `_info` to discover available activations
- Dynamically builds UI or routing tables

**Implementation:**
```rust
// plexus-core/src/plexus/plexus.rs:834-862
module.register_subscription(
    "_info",
    PLEXUS_NOTIF_METHOD,
    "_info_unsub",
    move |_params, pending, _ctx, _ext| {
        let info = json!({
            "backend": backend_name,
            "activations": [{
                "namespace": namespace,
                "version": version,
                "description": description
            }]
        });

        let info_stream = futures::stream::once(async move { info });
        pipe_stream_to_subscription(pending, wrap_stream(...)).await
    }
)
```

### Per-Transport Details

#### REST HTTP Gateway

**Exposed Methods:**
- ✅ `schema` - as regular endpoint
- ✅ `{method}.schema` - via suffix pattern
- ✅ All user methods - with appropriate HTTP verbs

**Not Exposed:**
- ❌ `.call`, `.hash`, `_info` - WebSocket-specific

**Routing Strategy:**
```rust
// src/http/bridge.rs:88-129
fn schemas_to_rest_routes<A>(schemas: Vec<PluginSchema>) -> Router {
    let registry = MethodRegistry::from_schemas(schemas);
    let mut router = Router::new();

    // Register each method (including "schema") with its HTTP method
    for method_info in registry.all_methods() {
        let path = format!("/{}/{}", method_info.namespace, method_info.method);

        let method_router = match method_info.http_method {
            HttpMethod::Get => get(rest_method_handler),
            HttpMethod::Post => post(rest_method_handler),
            HttpMethod::Put => put(rest_method_handler),
            HttpMethod::Delete => delete(rest_method_handler),
            HttpMethod::Patch => patch(rest_method_handler),
        };

        router = router.route(&path, method_router);
    }

    router.with_state(state)
}
```

**Key Insight:** REST treats `schema` as a regular method because it's in `PluginSchema.methods[]`.

---

#### MCP Gateway

**Exposed Methods:**
- ✅ All methods (including `schema`) as MCP tools

**Not Exposed:**
- ❌ `.call`, `.hash`, `_info` - not MCP concepts
- ❌ `{method}.schema` - use `{ns}.schema` with params instead

**Translation Strategy:**
```rust
// src/mcp/bridge.rs:38-71
fn schemas_to_rmcp_tools(schemas: Vec<PluginSchema>) -> Vec<Tool> {
    schemas
        .into_iter()
        .flat_map(|activation| {
            let namespace = activation.namespace.clone();

            // Convert each method (including "schema") to a tool
            activation.methods.into_iter().map(move |method| {
                Tool {
                    name: format!("{}.{}", namespace, method.name),
                    description: method.description.clone(),
                    input_schema: ensure_object_type(method.params),
                }
            })
        })
        .collect()
}
```

**MCP Client View:**
```json
{
  "tools": [
    {
      "name": "users.get_user",
      "description": "Get user by ID",
      "inputSchema": {"type": "object", "properties": {...}}
    },
    {
      "name": "users.create_user",
      "description": "Create new user",
      "inputSchema": {"type": "object", "properties": {...}}
    },
    {
      "name": "users.schema",
      "description": "Get plugin or method schema...",
      "inputSchema": {"type": "object", "properties": {"method": ...}}
    }
  ]
}
```

**Key Insight:** MCP sees `schema` as just another tool in `tools/list`.

---

#### WebSocket JSON-RPC

**Exposed Methods:**
- ✅ All 5 protocol methods
- ✅ All user methods (via `.call`)

**Subscription Registration:**
```rust
// plexus-core/src/plexus/plexus.rs:754-862
pub fn into_rpc(self) -> RpcModule<()> {
    let ns = self.namespace();
    let mut module = RpcModule::new(());

    // Per-activation subscriptions
    module.register_subscription("{ns}.call", ...);    // Generic
    module.register_subscription("{ns}.schema", ...);  // Discovery
    module.register_subscription("{ns}.hash", ...);    // Cache validation

    // Global subscription
    module.register_subscription("_info", ...);        // Backend info

    module
}
```

**Dual Access to `schema`:**
1. **Direct:** `users.schema` subscription
2. **Generic:** `users.call` with `method: "schema"`

Both work identically but provide different ergonomics:
```json
// Option 1: Direct subscription
{"method": "users.schema", "params": {"method": "get_user"}}

// Option 2: Via .call
{"method": "users.call", "params": {"method": "schema", "params": {"method": "get_user"}}}
```

**Key Insight:** WebSocket provides the richest protocol method support with specialized subscriptions.

## Implementation Details

### Schema Method Generation

The `schema` method is **auto-generated by the macro** and added to the methods array:

```rust
// plexus-macros/src/codegen/method_enum.rs:300-310
fn compute_method_schemas() -> Vec<MethodSchema> {
    // ... process user methods ...

    // Add the auto-generated schema method
    let schema_method = MethodSchema::new(
        "schema".to_string(),
        "Get plugin or method schema. Pass {\"method\": \"name\"} for a specific method.",
        "auto_schema".to_string(),
    );
    methods.push(schema_method);  // ← Now in PluginSchema.methods[]

    methods
}
```

**Why this works:**
1. Macro adds `schema` to `methods[]` during code generation
2. Transport bridges iterate `methods[]` without special-casing
3. Every transport gets schema discovery automatically

### Schema Method Dispatch

The `schema` method is **handled specially in the dispatch match**:

```rust
// plexus-macros/src/codegen/activation.rs:225-282
async fn call(&self, method: &str, params: Value) -> Result<PlexusStream, PlexusError> {
    match method {
        // User methods dispatch to actual async functions
        "get_user" => { /* call self.get_user(...) */ },
        "create_user" => { /* call self.create_user(...) */ },

        // Schema method has inline implementation
        "schema" => {
            let method_name: Option<String> = params.get("method")
                .and_then(|v| v.as_str())
                .map(|s| s.to_string());

            let plugin_schema = self.plugin_schema();

            let result = if let Some(ref name) = method_name {
                // Return specific method schema
                plugin_schema.methods.iter()
                    .find(|m| m.name == *name)
                    .map(|m| SchemaResult::Method(m.clone()))
                    .ok_or_else(|| PlexusError::MethodNotFound {
                        activation: namespace.to_string(),
                        method: name.clone(),
                    })?
            } else {
                // Return full plugin schema
                SchemaResult::Plugin(plugin_schema)
            };

            Ok(wrap_stream(
                futures::stream::once(async move { result }),
                concat!(namespace, ".schema"),
                vec![namespace.into()]
            ))
        },

        // Suffix pattern for {method}.schema
        _ if method.ends_with(".schema") => {
            let method_name = method.strip_suffix(".schema").unwrap();
            // Lookup and return MethodSchema
        },

        // Child routing or MethodNotFound
        _ => { /* ... */ }
    }
}
```

**Key Points:**
- No actual `async fn schema()` exists in user code
- Implementation is generated inline by macro
- Returns `SchemaResult` enum (Plugin or Method variant)

### Transport Bridge Patterns

#### REST HTTP Bridge

**Treats all methods uniformly:**

```rust
// src/http/bridge.rs:55-73
impl MethodRegistry {
    pub fn from_schemas(schemas: Vec<PluginSchema>) -> Self {
        let mut methods = HashMap::new();

        for schema in schemas {
            for method in schema.methods {  // "schema" is just another method
                let key = format!("{}.{}", schema.namespace, method.name);
                methods.insert(key, RestMethodInfo {
                    namespace: schema.namespace.clone(),
                    method: method.name.clone(),  // "schema" becomes a route
                    streaming: method.streaming,
                    http_method: method.http_method,  // POST (default)
                });
            }
        }

        Self { methods: Arc::new(methods) }
    }
}
```

**Result:** `POST /rest/users/schema` is registered automatically.

#### MCP Bridge

**Converts all methods to tools:**

```rust
// src/mcp/bridge.rs:229-244
async fn list_tools(&self) -> Result<ListToolsResult> {
    let schemas = if let Some(flat) = &self.flat_schemas {
        flat.as_ref().clone()
    } else {
        vec![self.activation.plugin_schema()]
    };

    let tools = schemas_to_rmcp_tools(schemas);  // "schema" → tool

    Ok(ListToolsResult {
        tools,
        next_cursor: None,
        meta: None,
    })
}

async fn call_tool(&self, name: &str, arguments: Value) -> Result<...> {
    let (namespace, method) = name.split_once('.').unwrap();

    // All tools (including "schema") go through activation.call()
    let stream = if let Some(route_fn) = &self.route_fn {
        route_fn(name.to_string(), arguments).await?
    } else {
        self.activation.call(method, arguments).await?
    };

    // Stream back as SSE
    Ok(CallToolResult { content: stream_to_content(stream).await })
}
```

## Design Rationale

### Why is `schema` in `methods[]`?

**Benefits:**
1. **Transport uniformity** - No special-casing in REST/MCP bridges
2. **Auto-discovery** - Clients see it in tools/endpoints list
3. **Self-describing** - Has params, description, http_method like user methods
4. **Introspection** - Can query schema for the schema method itself (meta!)

**Trade-offs:**
- Slightly confusing (no actual async fn in impl block)
- Must be handled specially in dispatch code
- Can't be removed/overridden by users

---

### Why WebSocket-only protocol methods?

**`.call`, `.hash`, `_info` are WebSocket-only because:**

1. **REST has direct routing:**
   - `POST /rest/users/get_user` already invokes the method
   - No need for generic `.call` wrapper

2. **MCP has `tools/call`:**
   - MCP protocol provides its own generic invocation
   - Don't need another layer of indirection

3. **WebSocket benefits from subscriptions:**
   - Single persistent connection
   - Efficient for repeated invocations
   - Streaming responses built into protocol

4. **Cache validation is WebSocket use case:**
   - Long-lived connections benefit from cache checking
   - REST/MCP typically short-lived (fetch-and-disconnect)

---

### Why not expose `_info` in REST/MCP?

**REST:**
- Could add `GET /rest/_info` endpoint
- Not implemented yet (future enhancement)
- OpenAPI/Swagger can provide similar discovery

**MCP:**
- MCP has its own initialization handshake
- Server capabilities negotiated during `initialize`
- `_info` would be redundant with MCP metadata

---

### Suffix Pattern Design

The `{method}.schema` suffix pattern provides **ergonomic method discovery**:

```bash
# Instead of:
POST /rest/users/schema
{"method": "get_user"}

# You can write:
POST /rest/users/get_user.schema
{}
```

**Benefits:**
- More RESTful (resource-oriented)
- No request body needed
- Self-documenting URLs

**Implementation:**
```rust
_ if method.ends_with(".schema") => {
    let method_name = method.strip_suffix(".schema").unwrap();
    let plugin_schema = self.plugin_schema();

    if let Some(m) = plugin_schema.methods.iter().find(|m| m.name == method_name) {
        let result = SchemaResult::Method(m.clone());
        return Ok(wrap_stream(...));
    }

    // Not a local method - fall through to child routing
}
```

**Key:** Checks local methods first, then falls through to child routing for hub activations.

## Conclusion

Protocol-level methods provide a **universal introspection layer** across all Plexus transports:

- **`schema`** is treated as a first-class method, appearing in `PluginSchema.methods[]`
- Transport bridges handle it **uniformly** alongside user methods
- WebSocket provides **additional protocol methods** (`.call`, `.hash`, `_info`) as specialized subscriptions
- Design enables **consistent discovery** regardless of transport choice

The key insight: **Protocol methods are just methods**, ensuring universal discoverability and consistent API surface across all transports.
